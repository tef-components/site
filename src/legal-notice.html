Open Source @ TID

This document will provide a guide, along with tools and procedures, to allow initiatives to open whichever software they want to release as Open Source.

Opening our code (at least the no-core-business parts) will allow us to increase our visibility and, if we are lucky, to obtain nice feedback and contributions from the external community.

How should we start?

First of all it’s necessary to define a procedure that states the responsibility and the ownership of the free code releasing:

  • - The Area Manager and Area TechLead will be the last responsible to decide which software artifacts should be Opened based upon Initiative Technical Leaders proposals.

  • - All the opened software releases must be licensed. We have chosen Affero GPLv3 as the most convenient license. If we want to be free, we should be as free as possible (this is the best way to protect our dual-private releases).

  • - The initiative technical leader (or a chosen delegate) will be the administrator/moderator of the open project.

  • - In order to guarantee the software quality Q&A will provide a checklist and style guide that should fulfill every software artifact (some mandatory, some desired). Anyway it is possible that some projects at “pre-alpha” phase reach the liberation phase, so the projects must clearly advise the community with maturity level and expectations of the released product.

  • - The last owner of the software artifact quality will be the technical initiative leader.

  • - From this point it will be necessary to contact with the epg@tid.es in order to add the selected persons with their corresponding permissions to the GitHub TID organization account. Remind all the Open code must be released in the TID organization account, so we can manage and monitor all our Open source from a single point.

  • - Those projects accepting external contributions from the community must create their own entry point to external collaborators so they can apply for contribution/commit privileges. For example providing an email ([project].Contributors@tid.es). Other kind of contributions (bug reporting, comments, community support may need other kind of entry points).

  • - For internal contributions please contact with the Initiative Tech Leader or the project owner asking for priviledges.

  • - EPG will help you as far as possible in the provision of mailing list, and contribution channels associated to the released project (wiki, blog...).

  • To contribute with code it will be mandatory that the external developer sign and accept the TID Contribution Policy (see later) that will be sent to her in response to her contributor apply.

  • The technical leader (or her designated person) has the responsibility to moderate every external code contribution. Accepting just those contributions providing a differentiate value to the software artifact and recoding those contributions that were valid but considered of small entity or impact. Recoding could be good, for example, to include a small patch. So we can retain the IPR and may no need to require any signed document.

Licencing

The main idea is to provide a dual licensing of the selected products. That is, provide an Open version for the community and maintain a private version for internal purposes. Remind that every software release may have it’s own license, so we can have as many as we want/need, as far as we maintain the IPR of the software.

So Affero GPLv3 (http://www.gnu.org/licenses/agpl-3.0.html) will be chosen for the Open releases. It “guarantees” (as far as possible) that any usage or modification of our products will remain free so we can take advantage of it. While if somebody wants to use our software in private/closed products they will need contact TID for another release with the right license agreement. The usage of GPL like licences may discourage the usage of our software, but remind that this is just a start.

Dealing with Contributions

Obtain useful contribution from the community is one of the tough parts of the Open Source experience. It is really difficult to get the community involved, but it could hoppefully happen and we must be ready.

There are different kinds of contribution we should expect. Code contribution must be fully aligned with our Contribution Policy. Right now it is a really restrictive one (just a painless start), and may discourage some contributors. But it is mandatory that TID maintain the IPR of the whole code (IPR could be shared also, depending on the contribution and the strategic value of the Open Project, so new contribution agreements could be designed for selected cases).

Other kind of contributions should be managed also: Bug reporting, comments, support, test cases and other quality attributes. Again the objective is to recognise the valuable contributions while maintaining the IPR of the product. For example some bug reports or comments would drive us to implement new requirements, so it is important that contributors give us the freedom of use their feedback as we need to (for example use it in our closed releases).

Anyway since it don’t seem to be a good idea to require signing the contribution agreement to all kind of contributions we could provide an statement, of implicit acceptance, dealing with this concerns in the front-page of the project (conservative approach) or, since this is a fine grain problem, we could handle and solve it on the fly, when valuable contributions, matching with this scenario appear.

I want to be open!

Open source projects need extra management. It’s not a matter of double-commit into repositories. If you get open for the community, you should offer a minimum level of service to that community. Surely It’s not going to be too much work, but If we offer a contribution channel we should provide a good service on it. For example answering to the project distribution list, moderating commits or comments, doing a FAQ, road-mapping public requirements,... It’s up to you which channels do you open.

Remind that the last objective to Open Code is to attract users and contributors, opening your code “just to be open” and without any kind of community management, and dissemination efforts is mostly useless.

Other important point is the quality of the opened sources (of course this must apply to all our software also). It’s recommended to review some comments, “spanish coding”, review the project structure, dependencies (and their licenses), provide examples and useful instructions to build, use and deploy the software... and refactor could be needed also. If you want to be open, do your best. You never know who is watching.

Using the Repository

We have generated a github “organization account” for TID Open Projects. It is a free account, so it can host only public projects. Every released Open Project must be aggregated to this organization account.

Take into account that some problems could arise if we use a different repository than GIT in our internal developments, including extra management and losing commit meta-information. So it is encouraged to use GIT for our Open Projects (in public and private releases).

The administration of the Organization Account will by done by the people of the EPG. So ask them if you want to be added.
